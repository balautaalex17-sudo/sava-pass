# fix: Mobile section-nav (dots) overlaps homepage content

## Summary

On phones, the immersive homepage's right-side section-nav — the vertical `01 02 03 04 05` numerals (`<nav class="dots">`) — floats over page content as you scroll: it sits on top of the event poster, the seats progress bar, the "70% sală plină" label, and crowds the "Cumpără bilet" button. Root cause: `.dots` is `position:fixed` with `mix-blend-mode:difference` and **no mobile-hide media query**, even though its sibling rail `.lrail` is already hidden at `≤760px`. The mobile experience pass ([web/docs/plans/2026-06-23-002-feat-mobile-experience-pass-plan.md](docs/plans/2026-06-23-002-feat-mobile-experience-pass-plan.md), commit `9cb707c`) hid the left rail but missed the right-side dots nav. The fix is one media-query rule, applied through the immersive extractor, plus a redeploy so the live Vercel URL reflects it.

This is a real bug in current code (not just a stale deploy) — `.dots` has no `≤Npx` hide in `web/app/_immersive/content.ts` today.

## Problem Frame

Evidence: 4 screenshots from the **deployed** app (`sava-pass.vercel.app`) on an Android phone at ~360px CSS width:
- **Shot 1 (intro):** fine. The floating ticket chips (`QR VALID`, `LOCURI 36`, `SCAN OK`, `13.5K RON`) are `.engine-ticket` decorative elements clipped by `.intro{overflow:hidden}` — intentional ambient bleed, **not** a bug.
- **Shot 2 (phone mockup → "Ce urmează" events):** the `01–05` dots float on the right, overlapping the white events section.
- **Shot 3 (event poster card):** the dots sit **on top of** the "Echoes Unplugged" poster image, numerals clipped against it.
- **Shot 4 (event detail + buy CTA):** the dots overlap the seats progress bar, the "70% sală plină" label, and reach into the "Cumpără bilet · 45 RON" button. `mix-blend-mode:difference` inverts the numerals against the content behind them (why `03` reads orange/blue over the poster).

The `.dots` nav is `aria-hidden="true"` and purely a desktop scroll affordance (click a number to jump to a section). On a 360px viewport there is no side margin to hold a fixed vertical rail, so it collides with content. Hiding it on mobile loses no function — mobile users scroll, and the rail is already decorative/aria-hidden. This mirrors how `.lrail` is already handled.

Relevant code (all in `web/app/_immersive/content.ts`, generated by `web/scripts/extract-immersive.mjs`):
- `.dots{position:fixed;top:50%;right:clamp(14px,2.4vw,30px);transform:translateY(-50%);z-index:55;...mix-blend-mode:difference;...}` — no mobile-hide.
- `.lrail` **is** hidden: `@media(max-width:760px){.lrail{display:none;}}`.
- Existing immersive breakpoints: `520, 560, 760, 820, 900, 920px`.

## Requirements

- **R1** — On phones (360–430px) and small tablets in portrait, the `01–05` section-nav does not overlap or visually interfere with any homepage content; it is hidden, matching how `.lrail` is already hidden.
- **R2** — Desktop (≥ the chosen breakpoint, e.g. 1280px) is unchanged: the dots nav still appears and behaves exactly as before.
- **R3** — The fix is visible on the live URL the user is testing (`sava-pass.vercel.app`), i.e. the homepage is rebuilt and redeployed.

## Key Technical Decisions

- **KTD1 — Hide `.dots` on mobile; do not reposition it.** On a 360px screen there is nowhere to place a fixed vertical rail without overlap, and the rail is `aria-hidden` decorative nav. Hiding is consistent with the already-hidden `.lrail`. Repositioning (e.g. moving to the bottom as a horizontal strip) is rejected as scope creep for a decorative affordance.
- **KTD2 — Match the `.lrail` breakpoint (`≤760px`) for consistency, verify the 760–920px band.** Hiding at `≤760px` covers every phone in the screenshots and keeps the two rails consistent. The one thing to confirm during implementation: between 760px and ~920px (tablet portrait), does `.dots` still overlap content (content may go closer to edge before 920px)? If so, bump the hide to `≤900px` or `≤920px` to align with where the immersive layout is already in single-column mode. This is the only open choice and is settled by looking, not guessing (see Open Questions).
- **KTD3 — Edit through the extractor, never hand-edit the generated file.** Per the established immersive workflow (002 KTD4 and repeated session lessons), immersive CSS changes are made either in the v3 source `Sava Pass #2/SavaPass Immersive v3.html` (the `@media(max-width:760px)` block next to `.lrail`) **or** as a `cssOut.replace()` patch in `web/scripts/extract-immersive.mjs`, then `node scripts/extract-immersive.mjs` regenerates `content.ts`. Do not hand-edit `web/app/_immersive/content.ts` as the source of truth — it is overwritten on the next extract. Prefer the extractor `cssOut` patch so the change survives re-syncs regardless of source-file state.
- **KTD4 — Verify via the served artifact, not just source.** Two session lessons apply: (1) a stale Turbopack `.next` cache has repeatedly served pre-edit CSS — after regenerating, confirm the served chunk actually contains the new `.dots` hide before concluding it works; (2) the screenshots are from Vercel, so a local fix is invisible to the user until redeploy (R3).

## Implementation Units

### U1. Hide the section-nav dots on mobile

**Goal:** Add a mobile-hide media rule for `.dots` so the section-nav no longer overlaps homepage content on phones, applied through the extractor.
**Requirements:** R1, R2.
**Dependencies:** none.
**Files:**
- `web/scripts/extract-immersive.mjs` — add a `cssOut` patch that injects `@media(max-width:760px){.dots{display:none;}}` (place it alongside / mirroring the existing `.lrail` hide). Alternatively edit the `@media(max-width:760px)` block in `Sava Pass #2/SavaPass Immersive v3.html` to include `.dots{display:none;}`; the extractor patch is preferred (KTD3).
- `web/app/_immersive/content.ts` — regenerated output of `node scripts/extract-immersive.mjs` (do not hand-edit as source).
**Approach:** Mirror the existing `.lrail` treatment. Confirm the regenerated `content.ts` contains the new rule. Decide the final breakpoint per KTD2 after checking the 760–920px band on the running app (default `≤760px`, bump to `≤900/920px` only if a gap is found). Desktop rule for `.dots` is untouched, so desktop behavior is unchanged.
**Patterns to follow:** the existing `@media(max-width:760px){.lrail{display:none;}}` rule in `content.ts`; the `cssOut.replace()` patch pattern in `extract-immersive.mjs` used for every prior immersive CSS change this session.
**Test scenarios:**
- Covers R1. Playwright mobile context at 360px and 390px on `/`: `.dots` computes `display === "none"` (and is therefore not in the layout); after scrolling through intro → hero → event → stats, no `.dots a` element's bounding rect intersects any content element (`.ev-poster`, `.ev-prog-bar`, `.btn-p`).
- Covers R2. Desktop context at 1280px on `/`: `.dots` computes `display !== "none"` and is visible on the right edge; clicking a dot still scrolls to its section (`#event`, `#stats`, etc.) — nav behavior preserved.
- The served CSS chunk for `/` contains `.dots{display:none}` inside a `max-width` media block (guards against the stale-`.next` cache: fetch the chunk URL from the page HTML and grep it).
- Zero new console errors on `/` after the change (mobile and desktop contexts).
**Verification:** On a 360/390px viewport the dots are gone and nothing overlaps the poster / seats bar / buy button; on desktop the dots are unchanged and still navigate; the served CSS chunk includes the rule; `npm run build` passes (kill the dev server first — both write `.next`).

### U2. Verify the homepage mobile scroll + redeploy

**Goal:** Confirm the homepage scroll is clean on representative phone widths (no other fixed-overlay overlap), then redeploy so the live URL the user tested reflects the fix.
**Requirements:** R1, R3.
**Dependencies:** U1.
**Files:**
- `web/scripts/verify-mobile.mjs` (new, optional but recommended) — a small Playwright script that loads `/` at 360 / 390 / 414px, scrolls through all sections via `window.__lenis.scrollTo(...)` (or incremental `mouse.wheel`, since `scrollTo` doesn't drive Lenis), and asserts: no horizontal overflow, `.dots`/`.lrail` not visible, no `position:fixed` overlay (`.dots`, `.tele`, `.rail`) intersecting text-bearing content, and zero console errors. This also closes the long-missing U7 verification from plan 002, scoped to the homepage.
- (deploy) Vercel CLI redeploy of the `web/` project (`sava-pass`, root dir `web`) — no file change; uses the existing token/link flow recorded in `projects/sava-pass/CLAUDE.md`.
**Approach:** Spot-check the other fixed/ambient overlays seen in the screenshots are correctly contained on mobile: `.tele` telemetry corners, `.rail` (the top 2px progress bar — fine, full-width), and confirm the `.engine-ticket` chips remain clipped by `.intro{overflow:hidden}` (decorative, expected). Then redeploy. Because the user is testing the deployed URL, the fix is not "done" from their side until the new deploy is live (and they hard-refresh, per the stale-cached-HTML lesson).
**Patterns to follow:** the Playwright skill scripts used this session (`window.__lenis.scrollTo` to drive Lenis, `mouse.wheel` increments, computed-style + `getBoundingClientRect` assertions, `pageerror` capture, `isMobile:true,hasTouch:true` context); the recorded Vercel CLI deploy sequence in `projects/sava-pass/CLAUDE.md`.
**Test scenarios:**
- `/` at 360 / 390 / 414px: no horizontal overflow on `document.documentElement`; `.dots` and `.lrail` not rendered; no fixed overlay rect intersects `.ev-poster` / `.ev-prog-bar` / `.btn-p` / headline text through a full scroll.
- After redeploy: the production page's served CSS chunk contains the `.dots` mobile-hide rule (proves the deploy carries the fix, not a stale build).
- `Test expectation: visual/responsive — verified via Playwright viewport assertions; this project has no unit-test harness.`
**Verification:** The homepage is clean at all three phone widths with no overlay overlaps and no console errors; the redeployed `sava-pass.vercel.app` serves the fix; `npm run build` passed before deploy.

## Risks & Dependencies

- **Breakpoint band (760–920px).** Default `≤760px` (matches `.lrail`) may leave a thin overlap on tablet-portrait if content reaches the edge before 920px. Mitigation: check that band in U1 and bump to `≤900/920px` if needed (KTD2). Low risk for the user's phone case (360px).
- **Stale Turbopack `.next` cache** has twice served pre-edit CSS this session. After regenerating, verify the served chunk contains the rule (or `rm -rf web/.next` + rebuild) before concluding it works (KTD4).
- **Live URL vs local.** The screenshots are from Vercel; the fix is invisible to the user until U2's redeploy, and they may need a hard refresh (cached HTML lesson). R3 depends on the redeploy step.
- **Extractor as source of truth.** Hand-editing `content.ts` would be lost on the next extract — the change must go through `extract-immersive.mjs` / the v3 source (KTD3).

## Open Questions (deferred to implementation)
